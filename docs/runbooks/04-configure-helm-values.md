# Runbook 04 - Configure Helm Values And Deployment Template

## Objective

Configure Helm to deploy the Auth Service container from Amazon ECR into Kubernetes.

This step connects:

Spring Boot Application
↓
Docker Image
↓
Amazon ECR
↓
Helm Chart
↓
Kubernetes Deployment

---

# Prerequisites

Verify:

```bash
docker images

aws ecr list-images \
--repository-name auth-service

helm version
```

Expected:

* Docker image exists
* ECR image exists
* Helm installed

---

# Step 1 - Review Deployment Template

Command:

```bash
cat helm/auth-service/templates/deployment.yaml
```

Purpose:

Inspect the Deployment template generated by Helm.

This file is not a Deployment yet.

It is a template that uses values from:

```text
values.yaml
```

to generate Kubernetes manifests.

---

# Step 2 - Understand Image Configuration

Template:

```yaml
image: "{{ .Values.image.repository }}:{{ .Values.image.tag }}"
```

Purpose:

Pull values from:

```yaml
image:
  repository:
  tag:
```

inside values.yaml.

Example:

```yaml
image:
  repository: 761018849945.dkr.ecr.ca-central-1.amazonaws.com/auth-service
  tag: "v1"
```

Generated Result:

```yaml
image: 761018849945.dkr.ecr.ca-central-1.amazonaws.com/auth-service:v1
```

---

# Step 3 - Understand Replica Count

Template:

```yaml
replicas: {{ .Values.replicaCount }}
```

Purpose:

Control how many application pods Kubernetes creates.

Example:

```yaml
replicaCount: 2
```

Generated Result:

```yaml
replicas: 2
```

Benefits:

* High Availability
* Fault Tolerance
* Load Distribution

---

# Step 4 - Understand Service Port

Template:

```yaml
containerPort: {{ .Values.service.port }}
```

Example:

```yaml
service:
  port: 8080
```

Generated Result:

```yaml
containerPort: 8080
```

This must match the Spring Boot application port.

---

# Step 5 - Understand Health Checks

Template:

```yaml
livenessProbe:
readinessProbe:
```

Purpose:

Allow Kubernetes to determine:

* Is the application healthy?
* Can it receive traffic?

Configured Endpoint:

```text
/health
```

Expected Response:

```json
{
  "status":"UP",
  "service":"auth-service"
}
```

---

# Step 6 - Understand Resource Requests

Example:

```yaml
resources:
  requests:
    cpu: "250m"
    memory: "256Mi"

  limits:
    cpu: "500m"
    memory: "512Mi"
```

Purpose:

Prevent a single application from consuming all node resources.

Requests:

Minimum guaranteed resources.

Limits:

Maximum allowed resources.

# Step 7 - Render Helm Template

Command:

```bash
helm template auth-service helm/auth-service
```

Purpose:

Render Helm templates locally.

No deployment occurs.

Helm generates Kubernetes YAML and prints it to the terminal.

This is the safest way to verify templates.

# Interview Questions

Q: What is deployment.yaml?

A: A Helm template that generates Kubernetes Deployment resources using values from values.yaml.

Q: Why use values.yaml?

A: To separate configuration from templates and make deployments reusable.

Q: What does replicaCount control?

A: The number of pod replicas deployed by Kubernetes.

Q: Why are liveness and readiness probes important?

A: They allow Kubernetes to determine whether an application is healthy and ready to receive traffic.

# Step 8 - Render Helm Templates

Command:

```bash
helm template auth-service helm/auth-service
```

Purpose:

Render Helm templates locally without deploying to Kubernetes.

This command converts:

* values.yaml
* deployment.yaml
* service.yaml
* hpa.yaml

into Kubernetes manifests.

Benefits:

* Detect syntax errors
* Verify image configuration
* Verify ports
* Verify resource limits
* Verify autoscaling

Expected Output:

```yaml
kind: Deployment

kind: Service

kind: HorizontalPodAutoscaler
```

Verification Checklist:

✓ Image URI points to ECR

✓ Port is 8080

✓ HPA created

✓ Resource requests configured

✓ Resource limits configured

✓ Service type is ClusterIP

Result:

Helm chart rendered successfully.
